## Summary

The core type system is unusually clean — `ActiveSet`, `ComplementOf`, `Warp<S>`, diverge/merge are already explicit contracts enforced by the compiler. The audit found issues at the edges: platform dispatch, dynamic input validation, handle provenance, and research module debt.

4 facet candidates (medium+ value). 0 critical correctness bugs. ~500 lines refactor opportunity.

## Facet Candidates

### 1. PlatformDispatch (CRITICAL)

**Contract:** GPU intrinsics (`GpuShuffle` trait) must provide semantically equivalent behavior across all compilation targets. CPU emulation must not hide bugs that would manifest on GPU.

**Origin:** both (observed in code duplication + prospected as forcing mechanism)
**Enforcement:** convention (CPU returns identity; GPU returns real shuffle; no validation of semantic equivalence)
**Value:** critical

**Evidence:**
- `src/gpu.rs:353-400` — CPU fallback returns identity for all 6 types (hides GPU-specific bugs)
- `src/gpu.rs:143-182` — AMD DPP intrinsics are stubs returning placeholders, no warning
- `src/cub.rs:109-125` — `exclusive_sum` lane-0 identity behavior differs CPU vs GPU
- `src/tile.rs:102-107` — tile shuffle CPU emulation is identity
- `src/gpu.rs:212-270` — NVIDIA impls hardcode `0xFFFFFFFF` mask, no connection to `ActiveSet::MASK`

**Unguarded boundary:** Code compiles and tests pass on CPU while actual GPU behavior may differ. The `reduce_sum` CPU emulation produces correct results (1+1+1...=32) by accident of the identity-shuffle math, but `inclusive_sum` does NOT produce correct prefix sums on CPU — it produces the same result as `reduce_sum`.

**Recommendation:** Formalize a `PlatformDispatch` facet that:
- Documents which operations have CPU/GPU semantic parity and which don't
- Gates scan-like operations behind `#[cfg(target_arch)]` or adds CPU multi-lane emulation
- Makes AMD stubs return `compile_error!` instead of silent identity

### 2. DynDivergeBounds (HIGH)

**Contract:** `diverge_dynamic(mask)` must produce masks that are valid for the current warp width. The complement invariant (`true | false == parent`) is structural, but the input mask is unvalidated.

**Origin:** prospected (forcing mechanism — ownership boundary without validation)
**Enforcement:** runtime-checked (debug_assert added post-audit)
**Value:** high (resolved)

**Evidence:**
- `src/dynamic.rs:127-133` — `diverge_dynamic` takes raw `u64`, no bounds check
- `src/dynamic.rs:46-49` — `DynDiverge` holds masks with no invariant proof
- `src/dynamic.rs:76-79` — `merge()` trusts construction without verification

**Scenario:** `warp.diverge_dynamic(0xFFFFFFFFFFFFFFFF)` on a 32-lane warp produces `true_mask = 0xFFFFFFFF` (correct) but `false_mask = 0xFFFFFFFF00000000` (32 phantom lanes). The `merge()` call succeeds but the masks are semantically invalid.

**Recommendation:** Add `debug_assert!(predicate_mask & !self.active_mask() == 0)` to `diverge_dynamic`. Document that masks wider than the warp width are clamped (current behavior) vs rejected (safer behavior).

### 3. WarpProvenance (MEDIUM)

**Contract:** `Warp<S>` handles must originate from `kernel_entry()` or `diverge_*()`. Forged handles break soundness.

**Origin:** prospected (trust boundary)
**Enforcement:** convention (`pub(crate)` on `Warp::new()`, but no sealed trait)
**Value:** medium

**Evidence:**
- `src/warp.rs:49-56` — `Warp::new()` is `pub(crate)`, not externally constructable
- `src/warp.rs:34-36` — `Warp<S>` is `PhantomData` only, no resource token
- `src/block.rs:156-162` — `BlockSession` follows same pattern

**Risk:** Low in practice — external code can't call `pub(crate)` constructors. But within the crate, any module can forge arbitrary warp handles. A sealed `WarpToken` pattern would make the invariant machine-checked.

**Recommendation:** Document the invariant explicitly. Consider sealing `ActiveSet` so external crates can't implement it for arbitrary types.

### 4. ResearchDebt (MEDIUM)

**Contract:** Research modules should import from core modules, not redefine traits and types.

**Origin:** observed (duplication across 9+ files)
**Enforcement:** convention (nothing prevents redefinition)
**Value:** medium

**Evidence:**
- `src/research/static_verify.rs:65-137` — Redefines ActiveSet trait + 10 type impls (~70 lines)
- `src/research/nested_diverge.rs:19-61` — Redefines ActiveSet + 9 types (~40 lines)
- `src/research/implicit_merge.rs:79-95` — Redefines ActiveSet + ComplementOf + 3 types (~15 lines)
- `src/research/shared.rs:37-164` — Exact copy of `block.rs` SharedRegion + WorkQueue (~130 lines)
- `src/research/crossbar_protocol.rs:53-92` — Parallel TileSet/TileComplement traits (~40 lines)
- `src/research/divergent_values.rs:36-40` — Redefines ActiveSet + ComplementOf (~5 lines)

**Total:** ~500 lines of duplication across 6+ research modules.

**Root cause:** Research modules were prototypes that predated the finalized core API. They define their own types to explore variations without coupling to the evolving core.

**Recommendation:** Low priority — research modules are gated behind `#[cfg(not(target_arch = "nvptx64"))]` and don't affect the shipped artifact. Clean up if/when research modules are promoted to public API.
