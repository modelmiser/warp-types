# Crystal Audit: Session-Typed GPU Divergence

**Date**: 2026-03-13
**Auditor**: Claude (mm-xtal-stem:audit)
**Scope**: Full paper (active/warp-types/paper/) — targeting PLDI/POPL/OOPSLA

## Summary

Four parallel agents analyzed principles, boundaries, assumptions, and interfaces. The audit surfaced 11 implicit assumptions, 2 critical. The most severe is a real implementation bug: `Warp<S>` derives `Copy+Clone`, which breaks the soundness proof's linearity requirement — a user can retain `Warp<All>` after diverging and shuffle on the stale handle. The second critical finding is that the "zero overhead" claim has never been verified on actual GPU hardware because no Rust→PTX compilation path exists.

**Facets found**: 11 candidates
**Confidence distribution**: 0 fundamental, 0 experimental, 4 model-predicted, 6 convention, 1 unguarded

## Facet Candidates

### 1. Linearity Violation

**Implicit contract**: Well-typed programs use warp handles linearly — diverge consumes the parent, merge consumes the children, no handle is used after consumption.
**Status**: observed
**Confidence**: unguarded — the implementation contradicts the claim
**Formalization value**: **critical**

**Evidence**:
- `src/warp.rs` derives `Copy, Clone` on `Warp<S>` (line 18)
- Paper §4.4 Lemmas 4.8-4.9 prove no-duplication and no-discard assuming linearity
- Paper §3.1: "Warp<S> is a linear type—it cannot be duplicated or discarded"
- `diverge_even_odd(self)` consumes `self`, but `Copy` means caller retains a copy
- Agents 2, 3, 4 all independently flagged this

**Current pattern**:
```rust
// This compiles but violates the soundness proof:
let w: Warp<All> = Warp::new();
let (e, o) = w.diverge_even_odd();  // consumes w... but Copy gives caller a copy
let bad = w.shuffle_xor(data, 1);    // uses the copy — shuffles with inactive lanes
```

**Assumptions** (with confidence):
- "Rust's ownership prevents warp reuse": unguarded — Copy overrides ownership
- "diverge(self) consuming self is sufficient": unguarded — only if !Copy

**Uncertainty**:
- Implementation vs theory: model — the theory is sound IF linearity holds; the implementation doesn't enforce it

**Formalization path**: code fix — remove `Copy, Clone` derives from `Warp<S>`. This is a 1-line fix that makes the implementation match the proof.
**Benefit**: Closes the gap between the soundness proof and the implementation. Without this fix, the paper's central theorem is not supported by the artifact.

### 2. Hardware Execution Fidelity

**Implicit contract**: The type system's safety guarantees and performance claims transfer from Rust/CPU to actual GPU execution.
**Status**: observed
**Confidence**: convention — the claim is standard for PL papers but unverified here
**Formalization value**: **critical**

**Evidence**:
- Paper §7.4: "Our type system prototype runs on CPU, emulating warp semantics. We have not generated PTX from our typed Rust code."
- Paper §6.2: "the generated code is identical to hand-written unsafe CUDA" — this is an illustrative claim, not compiler output
- Paper §7.2 comparison table: "Our type system | 0% | All paths | At compile time" — the 0% is for the CPU emulation
- No Rust→PTX compilation path exists (rust-cuda/accel experimental/abandoned)
- Agents 1, 2, 4 flagged this

**Current pattern**:
```
Paper claim: "Zero overhead by construction"
Reality: PhantomData erasure verified in Rust MIR. PTX/SASS output never examined.
Gap: merge_data() involves runtime select. const MASK materialized as immediate.
```

**Assumptions** (with confidence):
- "PhantomData erasure survives Rust→PTX": model-predicted — likely but untested
- "merge_data select is free on GPU": convention — predicated select is standard, but not verified
- "const MASK as immediate doesn't consume registers": model-predicted — depends on PTX compiler

**Uncertainty**:
- PTX compilation: epistemic — no data exists
- Register pressure impact: model — merge_data adds one predicated select per merge point

**Formalization path**: experimental-validation — compile to PTX via hypothetical Rust→GPU path, or manually translate a typed example to CUDA and compare PTX output.
**Benefit**: Would convert the strongest claim (zero overhead) from model-predicted to experimentally-verified.

### 3. Related Work Gaps

**Implicit contract**: The paper cites all relevant prior art for a top PL venue.
**Status**: observed
**Confidence**: convention — citation completeness is expected, not formally required
**Formalization value**: **high**

**Evidence**:
- **Ferrite** (Chen et al., ECOOP 2022): Session types embedded in Rust using PhantomData + traits — identical encoding technique. Not cited. Reviewers will ask "how does your encoding differ from Ferrite?"
- **Fault-tolerant MPST** (Viering et al., OOPSLA 2021; Adameit et al., CONCUR 2022): MPST with crash-stop failures. The paper's "quiescence" differs from crash-stop — should be explicitly distinguished.
- **LLVM uniformity/divergence analysis**: Compilers already track which values are uniform vs divergent. The paper should explain why a type system adds value beyond compiler analysis.
- **Asynchronous session subtyping undecidability** (Lange & Yoshida 2016): The paper's synchronous (lockstep) domain avoids this known obstacle. Should state this explicitly.

**Assumptions** (with confidence):
- "No prior work uses session types for GPU": experimentally-verified — web search confirms novelty
- "Ferrite is not relevant": unguarded — it uses the same Rust encoding for session types

**Formalization path**: documented-assumption — add citations and explicit comparisons to §8 Related Work.
**Benefit**: Preempts reviewer objections. Ferrite citation is near-mandatory for a PL venue.

### 4. Irreversible Control Flow

**Implicit contract**: All diverged paths eventually reach a merge point; control flow is well-bracketed.
**Status**: observed
**Confidence**: convention — the structured control flow assumption is standard for session types
**Formalization value**: **high**

**Evidence**:
- Paper §4.7: "Structured control flow: Diverge and merge are explicit operations"
- Paper does not discuss: early return, panic, break from diverged loop, __trap()
- `research/early_exit.rs` explores 4 exit patterns (ballot-based, reducing set, redistribution, existential tracking) — none appear in the paper
- If one branch returns, `Warp<S∩P>` is dropped — linearity violation (Lemma 4.9)
- Session types fundamentally assume well-bracketed communication

**Current pattern**:
```rust
let (active, inactive) = warp.diverge(|lane| predicate[lane]);
if some_condition {
    return early_result;  // drops `active` — linearity violation
}
// inactive handle is orphaned — can never merge back to Warp<All>
```

**Assumptions** (with confidence):
- "Programs don't need early exit from diverged code": convention — common in GPU code
- "Linearity rejection is the correct response": model-predicted — sound but may be too restrictive

**Formalization path**: documented-assumption — discuss in paper. The ballot-based exit pattern from research code is the principled workaround.
**Benefit**: Preempts "what about early return?" reviewer question. Shows awareness of the practical limitation.

### 5. Modern GPU Feature Gap

**Implicit contract**: The paper's scope (shuffle/ballot/reduce safety) covers the relevant GPU operations.
**Status**: observed
**Confidence**: convention — narrow scope is acceptable, but should be explicitly justified
**Formalization value**: **high**

**Evidence**:
- **Tensor cores** (WMMA/MMA): Warp-cooperative matrix operations distribute fragments across lanes. Diverged warps produce incorrect fragments — same bug class. Not mentioned.
- **Async copy** (cp.async, TMA): Warp-cooperative data movement. If lanes diverge between initiate and wait, arrival count is wrong. Not mentioned.
- **Dynamic parallelism**: Child kernel launches from device code. Type state of parent warp is unrelated to child warps. Not mentioned.
- **Cluster-level operations** (CC 9.0+): SM-to-SM communication. Not mentioned.
- Agents 2, 3 flagged this

**Assumptions** (with confidence):
- "Shuffle/ballot/reduce are the relevant operations": convention — they were in 2019; tensor cores dominate modern compute
- "Other operations are orthogonal": model-predicted — tensor core fragment correctness has the same structure

**Formalization path**: documented-assumption — state scope explicitly, note that tensor cores have analogous divergence issues.
**Benefit**: Shows the type system's principle extends beyond shuffle, even if the implementation doesn't cover it yet.

### 6. Cross-Domain Transfer Fidelity

**Implicit contract**: The "Beyond SIMT" analogies (§9.5) represent genuine structural transfers, not false parallels.
**Status**: observed
**Confidence**: model-predicted — some analogies are stronger than others
**Formalization value**: **medium**

**Evidence**:
- **FPGA (§9.6)**: Strong — TileGroup<S> ↔ Warp<S> demonstrated with working prototype. Stale-pipeline-register isomorphic to shuffle-from-inactive-lane. Lost: timing, clock domains.
- **Distributed systems**: Moderate — node quiescence maps to lane inactivity. Lost: Byzantine failure, network partitions, non-deterministic timing. SIMT is deterministic; distributed is not.
- **Database queries**: Weak — "active rows after predicate pushdown" is superficial. Lost: aggregation, joins, multi-table operations. No predictive power.
- **Proof search**: Weakest — case splits as sub-goals is high-level only. Lost: backtracking has no warp analog.
- Agent 4's false-comparability analysis: mechanism diverges for databases/proof

**Assumptions** (with confidence):
- "Participatory computation is a unifying concept": model-predicted — true at abstract level
- "Each analogy transfers actionable type-system techniques": unguarded for databases/proof

**Formalization path**: sensitivity-analysis — apply mm-xtal-stem:compare false-comparability check (mechanism/scale/coupling) to each claimed domain.
**Benefit**: Strengthens FPGA claim with explicit mechanism match; softens database/proof claims to "structural similarity" rather than "type system transfers."

### 7. Active Set Vocabulary Scaling

**Implicit contract**: The finite vocabulary of marker types (12 types, ~20 trait impls) is sufficient for practical GPU programs.
**Status**: observed
**Confidence**: model-predicted — sufficient for the paper's examples; untested at scale
**Formalization value**: **medium**

**Evidence**:
- `active_set.rs`: 12 marker types (All, Even, Odd, LowHalf, HighHalf, EvenLow, EvenHigh, OddLow, OddHigh, Lane0, NotLane0, None)
- 8 ComplementOf impls, 8 ComplementWithin impls — manually enumerated
- At depth 3, intersections like EvenLow ∩ (third predicate) don't exist as types
- At depth 4, need ~48+ types with ~100+ impls
- Paper §9.1 proposes proc macro generation — acknowledges the limitation

**Assumptions** (with confidence):
- "Depth 2-3 covers most practical divergence": convention — common GPU patterns are shallow
- "Proc macros solve the scaling problem": model-predicted — plausible but unimplemented

**Formalization path**: experimental-validation — survey real GPU kernels for max divergence nesting depth.
**Benefit**: Would validate (or invalidate) the convention that depth 2-3 is sufficient.

### 8. Warp Size Portability

**Implicit contract**: The type system is portable across GPU vendors and architectures.
**Status**: observed
**Confidence**: convention — portability is claimed but not demonstrated beyond 32/64
**Formalization value**: **medium**

**Evidence**:
- Paper §2.1: "32 for NVIDIA, 64 for AMD" — correct but incomplete
- AMD RDNA: Wave32/Wave64 dual mode — same GPU runs kernels at different warp sizes. Not mentioned.
- Intel Xe: Subgroup sizes 8, 16, or 32 (compiler-selected or explicit). Not mentioned.
- Vulkan subgroup operations: Portable but require runtime subgroup size query. Not mentioned.
- `warp_size.rs` explores const-generic parameterization — research, not integrated into core API
- Core `active_set.rs` uses `u32` masks — hardcoded to 32-lane

**Assumptions** (with confidence):
- "32/64 covers the relevant architectures": convention — Intel's 8/16 modes are increasingly common
- "u32 masks are sufficient": convention — breaks for AMD 64-lane without u64

**Formalization path**: documented-assumption — acknowledge Intel/Vulkan; note const-generic path.
**Benefit**: Strengthens the cross-vendor story for reviewers from AMD/Intel backgrounds.

### 9. Hardware Mask Divergence

**Implicit contract**: The type-level active set (e.g., Even::MASK = 0x55555555) matches the hardware's actual execution mask at runtime.
**Status**: observed
**Confidence**: model-predicted — true for structured control flow; may diverge on Volta+
**Formalization value**: **medium**

**Evidence**:
- Volta+ independent thread scheduling: threads can reconverge before the programmer's merge point
- `__activemask()` returns the ACTUAL hardware mask, which may differ from the type-level mask
- If threads reconverge early: active set is LARGER than type says — safe (conservative)
- If compiler moves code across diverge boundary: mask may differ — covered by "source-level safety" scope
- Paper §1: "the active set is a type-level property, not a runtime value" — correct but doesn't guarantee hardware agreement

**Assumptions** (with confidence):
- "Type-level mask = programmer intent = hardware mask": model-predicted — true for structured flow
- "Source-level safety is sufficient": convention — compiler bugs (LLVM#155682) show limits

**Formalization path**: documented-assumption — state explicitly that the type system models source-level semantics, not hardware scheduling.
**Benefit**: Preempts "but what about hardware reordering?" reviewer question.

### 10. DynWarp Merge Asymmetry

**Implicit contract**: The gradual typing path (DynWarp) provides the same safety invariants as the static path, just checked at runtime instead of compile time.
**Status**: observed
**Confidence**: model-predicted — the asymmetry is subtle
**Formalization value**: **medium**

**Evidence**:
- Static `merge()`: requires `ComplementOf<S1, S2>` — checks disjoint AND covering (S1∪S2=All)
- `DynWarp::merge()`: checks `self.mask & other.mask == 0` (disjoint) — does NOT check covering
- Two small disjoint sets (e.g., Lane0 + Lane1) can DynWarp::merge without ever reaching All
- The static path and dynamic path have different invariants — not just different enforcement times
- Agent 4 flagged this

**Assumptions** (with confidence):
- "DynWarp is a faithful runtime mirror of the static system": unguarded — it's weaker

**Formalization path**: code fix — add covering check to DynWarp::merge, or document the weaker invariant.
**Benefit**: Prevents users from assuming gradual migration preserves all static guarantees.
